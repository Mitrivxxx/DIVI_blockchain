SPRINT 3 – PODSUMOWANIE ZADAŃ

---

**KROK 1 — Zdefiniuj strukturę danych**

Minimalny model:
    - hash (bytes32)
    - issuer (address)
    - timestamp (uint256)
    - cid (string)
    - owner (address)
    - documentType (string)

Architektura:
    - hash jest kluczem głównym i nie dopuszczamy nadpisywania

---

**FAZA 2 — REGUŁY BIZNESOWE**

**KROK 2 — Kontrola dostępu**
    - Musisz jednoznacznie ustalić:
        - Kto może wywołać issueDocument? → tylko adresy z listy issuerów.
        - Gdzie przechowujesz listę? → mapping(address => bool) authorizedIssuers
        - Kto zarządza listą? → owner kontraktu.

**KROK 3 — Ochrona przed duplikatem**
    - Reguła: Jeśli hash istnieje → revert.
    - Sprawdzenie: require(documents[hash].timestamp == 0, "Document exists");
    - Dlaczego timestamp? Bo jeśli rekord nie istnieje → timestamp = 0.
    - To jest reguła integralności.

**KROK 4 — Walidacja danych**
    - Zanim zapiszesz:
        - hash != 0x0
        - owner != address(0)
        - bytes(cid).length > 0
        - bytes(documentType).length > 0

**FAZA 4 — PLAN TESTÓW (PRZED IMPLEMENTACJĄ)**

**KROK 7 — Testy pozytywne**
    - Musisz zapisać:
        - Autoryzowany issuer publikuje dokument
        - verifyDocument zwraca poprawne dane
        - timestamp jest ustawiony
    - Każdy test opisujesz:
        - kto wywołuje
        - co podaje
        - czego oczekujesz

**KROK 8 — Testy negatywne (ważniejsze)**
    - Musisz przetestować:
        - Nieautoryzowany issuer → revert
        - Ten sam hash drugi raz → revert
        - Pusty CID → revert
        - owner = zero address → revert
    - Jeśli nie potrafisz wymyślić testu łamiącego system, to system jest słaby.
