Sprint 2 — Implementacja backendu .NET + integracja IPFS
Cel: dodać warstwę backendu, która:

przyjmuje plik,
liczy hash SHA‑256,
wysyła plik do IPFS,
zwraca CID + hash,
przygotowuje dane do zapisu w smart kontrakcie.
1) Założenia Sprintu 2 (potwierdź)
Backend w .NET (Web API).
IPFS lokalny (np. Kubo) lub publiczny endpoint.
Brak autoryzacji (MVP) lub prosta autoryzacja instytucji?
2) Zadania funkcjonalne (krok po kroku)

2.2. Endpoint: Upload dokumentu
Wejście: plik PDF
Wyjście: hash, cid

Kroki:

Przyjmij plik jako IFormFile.
Odczytaj bytes i policz SHA‑256.
Wyślij bytes do IPFS.
Odbierz CID.
Zwróć JSON: { hash, cid }.
2.3. Serwis IPFS
Zaimplementuj klienta:
REST call do IPFS (/api/v0/add)
Obsłuż błędy:
timeout,
brak połączenia,
błąd serwera IPFS.
2.4. Serwis Hash
SHA256.Create() na byte[].
Zwróć hex lub bytes32 (do kompatybilności z Solidity).
2.5. Kontrakt – przygotowanie danych
Zapewnij zgodność:
bytes32 hash (32 bajty)
cid do przechowania off-chain (na razie zwracane w API)
Stwórz endpoint /prepare jeśli potrzebujesz:
konwersja hash do bytes32 do użycia w ethers.
3) Testy (MVP)
Unit tests:
hash z pliku ma stałą wartość.
błędy IPFS -> poprawne komunikaty.
Integration tests:
realny IPFS na localhost.
4) Dokumentacja techniczna
Dodaj do Documentation:

opis API endpointów,
schemat danych (hash, CID),
flow publikacji z IPFS (zgodny ze Sprint 1).
5) Kryteria zakończenia Sprintu 2

✅ Zwraca hash + cid
✅ IPFS działa
✅ Hash = SHA‑256
✅ Dokumentacja aktualna
